# 测试环境 Bug 修复

针对 **已部署到测试环境** 后暴露的问题进行修复与再发布，与 Bug 修复（`bug-fix.md`）类似但问题来源为 test/staging 环境。

## 输入

- 测试环境缺陷描述：用户反馈、监控、联调群、Issue 列表或 `e2e-report.md` 类文档在 test URL 上的复现
- 复现环境信息：test 域名、构建号、feature flag、账号角色
- 仓库中相关模块代码与配置（环境变量、API base URL 等）

## 输出

- 修复 commit（遵循外部规则中注入的 Git Commit 流程）
- 验证记录：在 test 环境复测通过说明，或更新的自动化报告
- 需要重新走发布时：**回到** 归档与发布（`archive-deploy.md`）推送分支、更新 MR 或按团队流程触发部署

---

## 执行步骤

1. **确认环境**：在测试环境复现问题，记录步骤与预期/实际行为（可附录到内部文档或回复线程）。
2. **定位与修复**：区分配置问题、环境差异、数据问题与代码缺陷；代码修复仍遵循 TDD 或小步验证。
3. **本地与 CI**：跑 lint / typecheck / test / build；必要时补充针对缺陷的回归测试。
4. **测试环境验证**：部署或等待流水线后，在 test URL 上再次验证；若仍有缺陷，重复 2–3。
5. **闭环发布**：修复确认后执行 `archive-deploy.md`（推送、更新 MR、触发部署等按项目约定）。

---

## 与 Bug 修复的差异

| 维度 | Bug 修复（`bug-fix.md`） | 测试环境 Bug 修复（`post-deploy-fix.md`） |
|------|------------------|------------------------|
| 问题来源 | `e2e-report.md` / 本地或预发浏览器验证 | 已上测试环境的真实调用与数据 |
| 回归入口 | 回到 `e2e-test.md` | 回到 `archive-deploy.md` 与部署流程 |
| 关注点 | 验收清单与 spec 对齐 | 环境配置、数据、权限与线上近似路径 |

---

## 护栏

- 不在未复现的情况下大面积回滚或重构
- 涉及 secrets、URL、租户数据时，不得在日志或 MR 描述中泄露敏感信息
- 修复后必须在 **测试环境** 实测，不得仅依赖本地通过
- 若缺陷根因是 **需求变更**，应同步更新 OpenSpec / spec 并走评审，而非静默改行为
