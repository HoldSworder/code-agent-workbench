# Bug 修复

根据 E2E 报告中的失败项逐项修复，并回归验证。

## 输入

- `{{openspec_path}}/e2e-report.md`（含失败场景、复现步骤、截图或日志引用）
- 相关源码与测试
- 可选：E2E 浏览器测试（`e2e-test.md`）使用的 `qa-*.md` / `specs/*/spec.md` 作为验收对照

## 输出

- 修复后的代码与（若适用）测试更新
- 对已修复项在报告或任务跟踪中标记处理结果
- 重新执行 **受影响** 的自动化用例或全量 E2E；全部就绪后 **回到** E2E 浏览器测试（`e2e-test.md`）直至验收结论通过或用户接受带债

---

## 执行步骤

1. **阅读** `e2e-report.md`，列出所有未通过项（按优先级：阻塞流程 > 数据错误 > UI 偏差）。
2. **逐条修复**：最小改动、可测试；必要时补充单元测试或契约测试。
3. **本地验证**：对改动模块跑相关 `pnpm test` / lint / typecheck；若报告项由浏览器发现，优先重跑对应浏览器场景。
4. **闭环**：修复批次结束后，执行 `e2e-test.md` 流程更新报告与验收结论。
5. 若引入新问题，重复 2–4，直到验收结论允许进入归档与发布（`archive-deploy.md`）或用户明确带债归档。

---

## 护栏

- 不得在未读 `e2e-report.md` 的情况下猜测性改代码
- 修复后必须重跑 **至少** 失败过的用例，不得仅凭主观判断「已好」
- 与 `qa-*.md` 冲突时，在报告或 commit 说明中显式记录决策
- 不跳过 E2E 浏览器测试（`e2e-test.md`）重新出报告就进入归档与发布（`archive-deploy.md`）（除非用户书面同意带债且已在报告中记录）
