# 故事点评估规则（默认模板）

> 评审中心服务（review-server）会把本文件 + 当前会话的 Spec 一起喂给 AI 评估器，
> 输出 frontend / backend / qa 三个角色的 points 与 rationale。
> 本文件可由各团队按实际节奏覆盖，**不要在这里写实现细节，只写「评估准则」**。

## 通用原则

- 1 故事点 ≈ 0.5 人日；最小粒度 0.5 点；最大不超过 8 点（超过应拆分）。
- 不评估"会议、确认、走读"等非编码工时；这些由 PM/SM 单独跟踪。
- 风险加成：复杂度 / 不确定性高时按 +1 ~ +2 点酌情上浮，并在 rationale 中说明。

## 角色分项

### 前端（frontend）

| 维度 | 0.5 ~ 1 点 | 2 ~ 3 点 | 5 点以上 |
| --- | --- | --- | --- |
| 页面 | 单一表单/列表，已有组件复用 | 新页面 + 联调 + 状态管理 | 多页面流程 / 大量交互 / 自定义可视化 |
| 接口 | ≤2 个简单接口 | 3~5 个，含错误态 | 6+ 接口或 SSE/WebSocket |
| 兼容性 | 仅桌面端 | 桌面 + 移动响应式 | 多端（含 Tauri/小程序）+ 多浏览器适配 |

### 后端（backend）

| 维度 | 0.5 ~ 1 点 | 2 ~ 3 点 | 5 点以上 |
| --- | --- | --- | --- |
| 接口 | 单接口、无新模型 | 新增 1 个领域模型 + CRUD | 跨模块编排 / 异步任务 / 幂等设计 |
| 数据 | 无 schema 变更 | 加字段 / 加索引 | 表结构调整 + 数据迁移 |
| 风险 | 无外部依赖 | 1 个第三方 / 内部服务 | 强依赖或新引入中间件（队列/缓存） |

### 测试（qa）

| 维度 | 0.5 ~ 1 点 | 2 ~ 3 点 | 5 点以上 |
| --- | --- | --- | --- |
| 用例 | 主流程 + 边界 ≤10 | 11~25 | 26+，或需要专项测试 |
| 自动化 | 复用现有用例 | 补充少量自动化 | 新建测试体系 / 性能或安全测试 |

## 输出约束

模型必须严格输出以下 JSON（无任何包裹）：

```json
{
  "frontend": { "points": 0, "rationale": "..." },
  "backend":  { "points": 0, "rationale": "..." },
  "qa":       { "points": 0, "rationale": "..." }
}
```

- 当某角色"不涉及"时，points 设为 0，并在 rationale 中明确说明原因（如"无前端改动"）。
- rationale 单段不超过 80 个汉字；只阐述维度命中点，不要重复评分原则。
