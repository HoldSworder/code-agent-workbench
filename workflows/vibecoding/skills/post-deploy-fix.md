# 测试环境 Bug 修复

针对 **已部署到测试环境** 后暴露的问题进行修复与再发布。

## 输入

- 测试环境缺陷描述：用户反馈、监控、联调群、Issue 列表
- 复现环境信息：test 域名、构建号、feature flag、账号角色
- 仓库中相关模块代码与配置

## 输出

- 修复 commit（遵循开发（`dev.md`）的 Git 流程：用户确认 → 审查 → commit）
- 验证记录：在 test 环境复测通过说明
- 需要重新走发布时：**回到** 发布（`deploy.md`）推送分支、更新 MR

---

## 执行步骤

1. **确认环境**：在测试环境复现问题，记录步骤与预期/实际行为。
2. **定位与修复**：区分配置问题、环境差异、数据问题与代码缺陷；代码修复仍遵循小步验证。
3. **本地验证**：跑代码 Review（`review.md`）中的 lint / typecheck / test / build；必要时补充针对缺陷的回归测试。
4. **测试环境验证**：部署或等待流水线后，在 test URL 上再次验证；若仍有缺陷，重复 2–3。
5. **闭环发布**：修复确认后执行 `deploy.md`（推送、更新 MR、触发部署等按项目约定）。

---

## 护栏

- 不在未复现的情况下大面积回滚或重构
- 涉及 secrets、URL、租户数据时，不得在日志或 MR 描述中泄露敏感信息
- 修复后必须在 **测试环境** 实测，不得仅依赖本地通过
