# 灰区讨论

在需求对齐（brainstorming）确认后、OpenSpec 落盘前，识别本次变更中尚未明确的实现细节（灰区），按维度向用户提问并收敛为可执行的决策表。

## 输入

- 经确认的解决方案与 API 契约（来自需求对齐阶段）
- 工程上下文摘要（用于「按项目默认」推断时的依据）
- 结构化需求摘要与 UI 规格约束（用于判断哪些维度 relevant）

## 输出

**决策表（decision table）**：按维度汇总用户选择或「项目默认」推断结果，字段清晰（维度、问题、选项、最终决定、备注），供 `fe-specflow:design-to-opsx` 写入 `proposal.md` 与 `specs/*/spec.md` 时引用。

### 灰区维度（仅讨论与本次变更相关的子集）

| 维度 | 关注点 |
|------|--------|
| UI 状态完整性 | 空状态、加载态、错误态、部分失败、禁用态 |
| 交互细节 | 表单验证时机、防重复提交、破坏性操作确认、数据更新策略 |
| 数据边界与展示 | 长文本截断、大数据量分页/虚拟列表、极端值、时间格式、数值精度 |
| 响应式与布局 | 目标断点、容器宽度、溢出策略 |
| 权限与条件渲染 | 角色差异、无权限处理、feature flag |
| 动效与过渡 | 列表增删动画、页面切换过渡 |

### 讨论方式

1. 自动识别涉及的 **2–4 个** 维度，避免全盘盘问
2. 每个问题须 **具体且带选项**（例如：「列表无数据时，复用现有 EmptyState 还是隐藏整块列表？」）
3. 用户可说「用项目默认」，Agent 基于仓库扫描到的既有模式推断并写入决策表
4. 将结论整理为决策表，传递给 design-to-opsx 阶段

### 跳过条件

- 变更仅为纯文案或样式微调，不涉及交互逻辑
- 用户明确表示跳过灰区讨论

跳过则产出简短说明与空表/最小表，不阻塞后续 Spec 落盘（前提是风险已被用户接受）。

## 护栏

- 禁止在未经用户确认（或「项目默认」合理推断）的情况下把关键灰区留到编码期再猜
- 推断「项目默认」时必须指向扫描到的具体模式，避免无依据统一口径
- 问题应一次聚焦一个决策点，避免冗长问卷降低完成率
