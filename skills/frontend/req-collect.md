# 需求采集

本阶段目标：得到 **完整、可评审** 的需求表述。来源可单一，也可多源组合；产出为结构化需求摘要，供设计与对齐使用。

## 输入

- 用户指定的需求来源（可多项）：飞书文档链接或标识、GitLab Spec 路径或 API 可访问标识、截图、对话中的纯文字、本地文件路径
- 用户是否声明了 **实现范围**（如仅某页面某区域）

## 输出

**结构化需求摘要（structured requirement summary）**，须满足：

1. 按来源逐一采集，提取需求点、业务规则、验收标准
2. 合并为统一视图，**每条需求标注出处**（飞书 / GitLab / 截图 / 文字 / 本地文件）
3. 来源冲突时 **不自行裁决**：在摘要中单独列出冲突点，留待 brainstorming 阶段请用户澄清
4. MCP、`GITLAB_TOKEN` 等不可用时须 **明确提示用户**，禁止静默跳过某来源

**范围限定**：用户指定范围时，该范围为必选实现边界；范围外内容仅作背景，不自动纳入实现清单。

## 护栏

- 禁止在凭证缺失时假装已拉取远端内容
- 禁止将用户未确认的范围外需求默认纳入本期交付
- 多源信息以可追溯（出处标注）为先，优于主观合并叙事
