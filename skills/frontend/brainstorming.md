# 需求对齐

在工程上下文、需求与设计约束齐备后，通过 Superpowers brainstorming 与用户对齐方案，确认前端实现路径与 API 契约。**本阶段不写 OpenSpec 文件、不写业务功能代码。**

**INVOKE SKILL: `superpowers:brainstorming`**

Agent 必须直接调用 Superpowers brainstorming 技能执行完整流程：探索项目上下文、逐问澄清（一次一个问题）、提出 2–3 个方案及权衡、逐节展示设计并获取用户确认。

## 输入

- 仓库扫描产出的 **工程上下文摘要**（技术栈、目录与模块、编码模式、OpenSpec 历史）
- 需求采集产出的 **结构化需求摘要**（含出处与待澄清冲突）
- 设计稿阶段产出的 **UI 规格约束**（若有）；局部变更须明确「本次改动边界」与「不改动区域」

## 输出

**经用户确认的解决方案**，须包含：

1. 与现有架构一致的 **前端实现要点**：组件拆分、状态管理、路由、样式方案；须优先复用现有公共组件与工具函数
2. **前端视角的 API 契约**：请求参数、响应结构、错误码等，足以支撑 mock 开发与联调对齐
3. 与 brainstorming 流程一致的确认记录（方案选择与用户确认点）

**禁止行为：**

- **不调用** `superpowers:writing-plans`（计划阶段由工作流其他部分接管）
- **不写入** `docs/superpowers/specs/`（OpenSpec 落盘由 `fe-specflow:design-to-opsx` 在后续阶段处理）

## 护栏

- **禁止** 跳过本技能或用手写提纲代替完整 brainstorming 流程
- **禁止** 在设计探索阶段编写业务功能代码
- KISS / YAGNI：方案与契约以可交付、可验证为界，避免过度设计
- 冲突与灰区未决项应显式列出，供下一阶段（灰区讨论或 Spec 落盘）承接
